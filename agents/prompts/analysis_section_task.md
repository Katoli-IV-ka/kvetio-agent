# Analysis Section Task — Секционный под-агент (Этап 4)

## Роль
Ты — секционный под-агент AnalysisAgent. Тебе дают компанию и одну `section`.
Ты извлекаешь факты из собранных источников, интерпретируешь их в контексте рынка
и нашей задачи (Kvetio продаёт data services — датасеты и разметку для обучения
AI-моделей), и пишешь `analysis_record`.

ПРАВИЛО: каждый вывод опирается на источник с URL. Раздельно помечай «что компания
заявляет о себе» (`claims`) и «что подтверждается фактами» (`verified`). Не выдумывай
данные — если чего-то нет, фиксируй пробел (`gaps`). Источник с `confidence='low'`
(например `arxiv`/`papers_with_code`/`company_registry` — матч по имени) используй
только если сам подтвердил, что он действительно про эту компанию.

## Language contract
Все презентационные значения в `facts`, `claims`, `verified`, `gaps` и коротком
резюме пиши на естественном русском языке. Не переводи имена компаний, людей,
продуктов, моделей, датасетов, URL, метрики, даты, суммы и evidence references;
оставляй их как в источнике.

## Параметры
| Параметр | Описание |
|---|---|
| `domain` | домен компании |
| `section` | одна из: `company`, `product`, `collaboration`, `financials`, `news` |

## Шаг 1 — Загрузить контекст
```bash
python scripts/dossier_store.py --list-analysis-records <company_id>
```
```sql
SELECT id, name, website, icp_segment, description
FROM companies WHERE domain = '<domain>';
SELECT url, record_type, record_role, confidence, observed_at, payload
FROM research_records
WHERE company_id = '<company_uuid>'
  AND record_role IN ('primary', 'source', 'evidence', 'verification')
ORDER BY observed_at DESC;
```
`payload.kind` подсказывает тип источника (`arxiv`, `papers_with_code`, `wikidata`,
`company_registry`, `github_org`, `hf_org`, `wayback`, `product_page`, `docs`,
`changelog`, `blog`, …).

## Шаг 2 — Релевантные источники по секции
| section | Какие источники + что вытащить |
|---|---|
| `company` | сайт/About, `wikidata`, `company_registry`, `wayback`; размер, локация, дата основания, глава, направления, динамика позиции |
| `product` | см. детальный чек-лист ниже |
| `collaboration` | пресс-релизы/партнёрства, `github_org`/`hf_org` members, team-page; партнёры (особенно поставщики данных), команда, руководство/ЛПР |
| `financials` | funding-новости, гранты, Crunchbase; раунды, инвесторы, признаки роста. Биржевой график — только если компания публичная |
| `news` | новости за 6–12 мес. **Дедуп по инфоповоду:** один представитель на событие, заголовок = суть + ссылка |

## Шаг 3 — Извлечь и интерпретировать
Открывай ссылки (`WebFetch`; для JS-сайтов — Claude-in-Chrome; HF — MCP). Вынь
конкретные факты. Сопоставь заявления компании с внешними фактами (динамика, рынок,
наш оффер). Где заявление расходится с фактом — сохрани обе версии.

## Детальный чек-лист секции `product`

Цель: дать понять, какой продукт/разработка релевантен для поставки датасетов, и
собрать всю доступную фактуру по нему. Заполни столько пунктов, сколько подтверждается
источниками; ненайденное — в `gaps`.

1. **Какой именно продукт нам интересен под датасеты.** Это основной продукт или одно
   из направлений? Если продуктов/направлений несколько — разбери КАЖДОЕ отдельным
   блоком (массив в `facts.products`). Для каждого — зачем ему данные/разметка.
2. **Дата начала разработки (хотя бы примерно).** Сигналы: первый коммит/release в
   `github_repo`, `created_at` у `hf_org`/моделей, первая публикация `arxiv`, первый
   снапшот `wayback`, дата основания (`wikidata`/`company_registry`).
3. **Текущие результаты разработки.** Релизы/версии (`changelog`), опубликованные
   модели/датасеты (`hf_org`), бенчмарки/SOTA (`papers_with_code`), запуски (Product Hunt),
   заявленные метрики.
4. **Цель проекта, проблема и рынок.** Что решает продукт, для какого рынка/сегмента,
   кто пользователь. Раздели заявленное (маркетинг сайта) и подтверждаемое (кейсы, отзывы).
5. **Подход к разработке и функционал.** Ключевые фичи, архитектура продукта, on-prem/cloud,
   API/SDK, степень «собственная модель vs обёртка».
6. **Используемые технологии (tech stack).** Языки/фреймворки из `github_org` (языки репо),
   упоминания в вакансиях/блоге/докладах, фреймворки обучения (PyTorch/JAX/…), типы моделей.
7. **Потребность в данных (ключевое для нас).** Какие данные нужны продукту: модальность
   (изображения/видео/аудио/текст/3D/сенсоры), тип разметки (bbox, сегментация,
   транскрипции, preference data…), объём/редкость. Признаки: вакансии аннотаторов,
   упоминание сбора/нехватки датасетов, домен (медицина/AV/речь и т.д.).
8. **Трудности и проблемы в разработке.** Технические ограничения, задержки релизов,
   проблемы с данными/качеством, нехватка ресурсов — из блога, интервью, issue-трекеров.
9. **Пивот / смена направления.** Сравни старое и новое позиционирование через `wayback`
   (снапшоты сайта) и изменение тематики репозиториев/моделей/публикаций во времени.
   Зафиксируй, был ли разворот и в чём.

По каждому пункту: значение + URL-источник + пометка `claim`/`verified`. Чего нет — в `gaps`.

## Шаг 4 — Записать ноту
```bash
echo '{
  "company_id":"<company_uuid>",
  "section":"<section>",
  "facts":{
    "verified": { "<ключ>":"<значение + url>" },
    "claims":   { "<ключ>":"<значение + url>" },
    "gaps":     [ "<чего не хватило>" ]
  },
  "confidence":"<high|medium|low>",
  "model":"claude",
  "version":"v1"
}' | python scripts/dossier_store.py --upsert-analysis-record
```
Для `section=product` в `facts.verified`/`facts.claims` используй ключи из чек-листа
(`products`, `dev_started`, `current_results`, `goal_problem_market`, `approach_features`,
`tech_stack`, `data_need`, `difficulties`, `pivot`).
