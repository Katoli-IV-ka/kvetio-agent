# Agent Upgrade — Design Spec

**Дата:** 2026-06-21
**Статус:** Draft for review
**Автор:** дизайн-сессия (brainstorming)
**Связано с:** `ARCHITECTURE.md`, `docs/AGENT-ECOSYSTEM.md`, `agents/prompts/*`

---

## 1. Цель

Главная цель пайплайна: находить максимально релевантные компании для продажи
data services (датасеты и разметка) и для каждой собирать достоверное, проверенное,
актуальное досье + релевантные точки контакта (ЛПР, принимающие решение о покупке
датасетов).

Эта доработка усиливает агентов и расширяет источники так, чтобы к этапу
формирования досье (Этап 4) вся необходимая информация уже была собрана,
подтверждена фактами и снабжена ссылками. Используются **только бесплатные и
общедоступные источники**.

Два приоритета этой итерации:
1. **Контакты / ЛПР** — менеджмент и лица, принимающие решение о покупке датасетов.
2. **Финансы** — сигналы покупательной способности и тайминга («есть деньги,
   возможно готовы покупать сейчас») + оценка финансового положения.

Структура итогового досье (6 разделов: о компании, продукт, сотрудничество,
финансы, новости, комплексный аудит) **сохраняется без изменений** — мы наполняем
её достовернее и глубже, а не переделываем.

## 2. Журнал ключевых решений

Зафиксировано в ходе дизайн-сессии:

1. **Масштаб = B.** Углубляем агентов И меняем поток (не только промпты).
2. **Таксономия ЛПР (tier 1–3)** принята как есть:
   - Tier 1 — экономические покупатели / держатели бюджета: Founder/CEO/CTO,
     VP/Head of Data, CDO, VP/Head of ML/AI.
   - Tier 2 — технические инициаторы потребности: Head of ML Eng, ML/Research Lead,
     Head of Perception/Autonomy, Data Engineering Lead.
   - Tier 3 — исполнители / тёплый вход: senior ML/Data engineers, Research
     Scientists, MLOps.
   - Отдельно — организационные каналы (общий инбокс, форма, Discord/Slack, партнёрский email).
3. **Минимум годного контакта = вариант C.** Собираем всех ЛПР (имя + роль + tier +
   хотя бы один канал), email не обязателен; «готовность к контакту» — производная
   от наличия прямого канала.
4. **Финансы = сигналы + оценка, не график.** Главное — факт недавнего/готовящегося
   финансирования или гранта и оценка финположения. Биржевой график — опционально,
   только для публичных.
5. **Никаких субъективных оценок в виде новых полей.** Новое поле может содержать
   только конкретный измеримый показатель (дата, ссылка), а не суждение. Субъективный
   синтез живёт только в нарративе раздела Аудит.
6. **Дата-провайдеры / «конкуренты» не отсеиваются.** Им самим периодически нужны
   большие датасеты → это отдельная категория лида (партнёрский трек), а не
   `not_relevant`. Реализуется статусом `data_partner`.
7. **Минимум новых полей.** Tier и обоснование ЛПР — в существующем `contacts.info`.
   Дата последнего финансирования — в существующем `dossiers.funding_date`; ссылка на
   сигнал — в `dossiers.extra_facts`. Новых колонок почти нет.

## 3. Перестроенный поток пайплайна

Новое выделено `★`.

```text
Discovery → Relevance → SourceExpansion + Enrichment
   → ★Contacts (ЛПР)         ← поднят в основной поток как обязательная стадия
   → Analysis (5 секций + аудит)
   → ★Verification (гейт качества)   ← новая стадия перед досье
   → Conclusions (досье)

Monitor — refresh-петля (как сейчас), при новом сигнале триггерит
          точечный ре-анализ только затронутых секций.
```

**Почему Contacts до Analysis.** Раздел «Сотрудничество» требует команду разработки
и руководство/ЛПР. Если контакты собраны заранее, аналитик их использует, а не
оставляет пробел.

**Почему Verification отдельной стадией, а не внутри Analysis.** Анализ и проверка —
разные режимы мышления; смешивая, теряем строгость. Отдельный гейт даёт явную точку
отсечения несвежих/неподтверждённых данных перед досье.

**Отвергнутая альтернатива:** вшить верификацию в Analysis — отклонено по причине выше.

## 4. Карта источников (только бесплатные/публичные)

Принцип: детерминированные факты (даты, суммы, тикеры, регистрационные данные) —
**скриптами-резолверами** (повторяемо, дедуп по `dedupe_key`); мягкие факты и
интерпретация — **агентом** через WebFetch/WebSearch.

| Раздел досье | Источник | Что даёт | Как берём | Статус сейчас |
|---|---|---|---|---|
| Финансы | SEC EDGAR (Form D) | частные раунды US: сумма, дата, инвесторы | резолвер | выкл → включить |
| Финансы | NIH RePORTER + EU CORDIS + SBIR.gov | гранты: сумма, дата | резолвер (`gov_grants`) | выкл → включить |
| Финансы | GDELT | funding-новости, дедуп инфоповодов | резолвер | выкл → включить |
| Финансы | Crunchbase (публичная страница) | базовые раунды / valuation | WebFetch (не платный API) | новый |
| Финансы | признаки подготовки к раунду | найм CFO/всплеск вакансий, демо-дни, цитаты «we're raising» | агент | новый |
| Финансы | Stooq CSV / Yahoo public | котировки год/полгода (только публичные) | резолвер (`market_data`) | новый, опц. |
| Контакты | arXiv PDF | почты авторов прямо в статьях | резолвер | новый |
| Контакты | GitHub | commit-emails + org members | резолвер (усилить `dm_github.py`) | есть |
| Контакты | HuggingFace org members | участники команды | резолвер | есть |
| Контакты | team/about-страницы | команда, руководство | резолвер/агент | есть |
| Контакты | Wellfound | основатели | резолвер | есть |
| Контакты | конференции (NeurIPS/CVPR/ICRA…) + личные сайты | спикеры, каналы | агент | новый |
| Контакты | email pattern (`{first}@domain`) | угадывание email без Hunter | резолвер | новый |
| Компания | Wikidata + OpenCorporates | дата основания, юрлицо, локация, размер | резолвер | выкл/новый |
| Компания | Wayback | динамика позиционирования | резолвер | есть |
| Продукт | Papers with Code + arXiv | R&D, технологии, направление | резолвер | выкл → включить |
| Продукт | GitHub languages | tech stack | резолвер | есть |
| Продукт | Wayback (diff) | детект пивота (старое vs новое позиционирование) | агент | новый |
| Новости | GDELT DOC API + Google News RSS + блог-RSS | новости с дедупом по событию | резолвер | выкл → включить |

**Платные стабы оставляем выключенными** (`linkedin`, `crunchbase` API, `similarweb`,
Apollo, Hunter) — отметка «будущее, если появится бюджет».

## 5. Изменения по агентам

- **Discovery** — в `payload` первичных записей класть «зацепки» для резолверов
  (тикер, EDGAR CIK, arXiv-автор), чтобы ускорить нижние стадии. Логика не меняется.

- **Relevance** — добавляет третий исход: компании, которые сами являются
  дата-провайдерами/«конкурентами», получают статус **`data_partner`** (а не
  `not_relevant`). Найденные тикер/CIK/HF-org фиксируются как `verification`-записи.
  В `not_relevant` уходят только реально не-наши (мёртвый сайт, не-AI, чистый
  интегратор/обёртка).

- **SourceExpansion + Enrichment** — подключают новые резолверы (Раздел 4). Чёткое
  разделение: Enrichment гоняет детерминированные скрипты (EDGAR, gov_grants, GDELT,
  Wikidata, Papers with Code, arXiv); SourceExpansion добирает мягкое агентом. Оба
  пишут только URL + payload, без интерпретации.

- **★Contacts (новая обязательная стадия, бывший DMEnrich)** — ищет ЛПР по таксономии
  tier 1–3; тянет каналы из расширенного набора (arXiv-почты, GitHub-почты, HF,
  team-pages, Wellfound, конференции); в `contacts.info` пишет `tier` и короткое
  обоснование «почему ЛПР по датасетам». Дедуп `(company_id, contact_type, name)`.
  Обрабатывает компании со статусом `relevant` И `data_partner`.

- **Analysis (секции)** — `analysis_section_task.md` переписывается подробно. Для
  каждой из 5 секций — чек-лист обязательных полей из ТЗ:
  - *О компании:* размер, локация, главное/второстепенные направления, дата основания,
    глава, текущее положение и динамика позиции за последние годы.
  - *Продукт:* какой продукт интересует под датасеты (основной или одно из направлений;
    если несколько — по каждому), дата старта разработки, текущие результаты, цель/
    проблема/рынок, подход и функционал, технологии, трудности, пивоты.
  - *Сотрудничество:* партнёры (особенно поставщики данных), команда разработки,
    руководство/ЛПР (из `contacts`).
  - *Финансы:* сигналы покупательной способности (раунд/грант, признаки подготовки к
    раунду, найм), оценка/valuation; график — только для публичных.
  - *Новости:* разные инфоповоды за 6–12 мес без повторов, заголовок = суть + ссылка.

  Жёсткие правила: «заявление компании» и «подтверждённый факт» помечаются раздельно;
  каждый вывод несёт URL; пробел фиксируется как пробел, не выдумывается.

- **Analysis (аудит)** — добавляет явный синтез по ключевому показателю: есть ли
  деньги + подходящий ли момент + есть ли потребность в данных → нарративный вывод +
  конкретный угол оффера + по каким именно ЛПР заходить (ссылка на собранные контакты).
  Для `data_partner` — партнёрский угол вместо прямой продажи.

- **★Verification (новая стадия)** — гейт перед досье. Проверяет свежесть (пороги в
  `config`), живость ссылок, конфликты claim↔fact. Помечает факты
  `verified / unverified / stale`. В досье как факт идёт только `verified`.

- **Conclusions** — собирает 6 секций + блок ЛПР (top-контакты с tier) + дату
  последнего финансирования (`funding_date`) + ссылку на сигнал (`extra_facts`).
  `unverified`/`stale` выносит в отдельный список «требует проверки», не как факты.
  Для `data_partner` формулирует партнёрский угол.

- **Monitor** — при новом сигнале (раунд, грант, всплеск найма) триггерит ре-анализ
  только затронутых секций (financials/news), а не всю компанию.

## 6. Модель данных (минимальные обратносовместимые правки)

Цель — почти не плодить колонок; детали живут в нарративе и `research_records`.

- **`companies.status`** — добавить значение `data_partner` в CHECK-констрейнт.
  Downstream-стадии (SourceExpansion, Enrichment, Contacts, Analysis, Conclusions)
  выбирают `relevant` И `data_partner`.
  - *Нюанс:* статус в пайплайне только повышается. Чтобы партнёрская
    принадлежность не потерялась при продвижении (sources_gathered → analyzed → …),
    Relevance дополнительно пишет durable `research_record` с типом
    `data_partner_flag`. Downstream-агенты и Аудит определяют партнёрский трек по
    этому record, а не по текущему значению статуса. Так статус `data_partner`
    работает как первичный сигнал релевантности, а флаг-запись — как устойчивая
    память. См. открытый вопрос в Разделе 11.
- **`contacts`** — новых колонок нет. `tier` и обоснование ЛПР пишутся текстом в
  существующее `info`. «Готовность к контакту» — производная от наличия прямого
  канала (email/личный сайт/X/LinkedIn), не хранится.
- **`dossiers`** — новых колонок нет. `funding_date` = дата последнего события
  финансирования (неважно, грант или инвестиция). Ссылка на сигнал-источник → в
  `extra_facts` (например, `{"last_funding_source_url": "..."}`). Valuation, признаки
  подготовки к раунду, число вакансий — в нарративе раздела Финансы и в
  `research_records`.
- **`research_records.record_type`** — новые типы в словаре `record_types` (+ в
  `models.py::SignalType`): `form_d`, `grant`, `arxiv_paper`, `quote`, `job_count`,
  `market_quote`.
- **Статус «достоверности» факта** — выражается на уровне `research_records`
  (`verified/unverified/stale` в `payload` или `confidence`), используется
  Verification и Conclusions. Без новой таблицы.

> Примечание по схеме. В репозитории есть два описания финального слоя: действующая
> `dossiers` (`ARCHITECTURE.md`) и предложенная `company_profiles` (спек 2026-06-19,
> помечен SUPERSEDED). Этот дизайн опирается на действующую `dossiers`. Если переход
> на `company_profiles` всё же состоится — те же поля-сигналы кладутся туда.

## 7. Правила качества данных

- **Пороги свежести (в `config`, не в промпте):**
  - Новости — событие ≤ 12 мес (приоритет ≤ 6).
  - Финансовый сигнал «есть деньги сейчас» — раунд/грант ≤ 18 мес считается актуальным;
    старше → `stale`, не идёт в шапку как текущий сигнал.
- **Живость ссылок:** Verification дёргает HEAD/GET; 404 / редирект на главную /
  только-Wayback → факт помечается `unverified`.
- **Claim vs fact:** при противоречии заявления компании и внешнего факта сохраняются
  обе версии; расхождение выносится в Аудит.
- **Дедуп новостей:** один представитель на инфоповод (кластеризация по сущности+дате
  через GDELT/заголовок), заголовок = суть + ссылка.
- **Провенанс:** каждый вывод в досье ссылается на `research_record` через
  `analysis_links` / `dossier_links`. Усиление: нет ссылки → факт не публикуется.

## 8. Матрица «данные досье → источник → агент → верификация»

| Поле/данные | Раздел | Источник(и) | Кто собирает | Верификация |
|---|---|---|---|---|
| Размер, локация, дата основания, глава | Компания | OpenCorporates, Wikidata, сайт/About | Enrichment + Analysis | дата осн. сверяется ≥2 источника |
| Направления, позиция и динамика | Компания | сайт, Wayback, новости | Analysis | Wayback diff подтверждает динамику |
| Продукт(ы), дата старта, результаты | Продукт | сайт/docs/blog, GitHub, HF | Analysis | дата старта из repo/HF created_at |
| Цель/проблема/рынок, технологии, пивот | Продукт | сайт, arXiv, Papers w/Code, GitHub langs, Wayback | Analysis | tech подтверждается repo/публикацией |
| Поставщики данных, партнёры | Сотрудничество | пресс-релизы, новости, сайт | SourceExpansion + Analysis | ссылка на анонс |
| Команда разработки | Сотрудничество | GitHub/HF members, team-page, arXiv | Contacts | ≥1 публичный профиль |
| Руководство / ЛПР (tier) | Сотрудничество | team-page, Wellfound, LinkedIn public, конф. | Contacts | роль подтверждена источником |
| Последнее финансирование (дата) | Финансы | SEC EDGAR, gov_grants, GDELT, Crunchbase public | Enrichment | ссылка-сигнал в `extra_facts` |
| Признаки подготовки к раунду | Финансы | вакансии (ATS), демо-дни, цитаты | Enrichment + Analysis | прямая цитата + URL |
| Оценка / valuation | Финансы | Crunchbase public, новости | Analysis | ≥1 источник, помечается как оценка |
| Котировки (только публичные) | Финансы | Stooq/Yahoo | Enrichment | тикер сверен с EDGAR/Wikidata |
| Новости по разным инфоповодам | Новости | GDELT DOC, Google News RSS, блог-RSS | Enrichment + Analysis | дедуп + дата ≤ 12 мес |
| Комплексный аудит | Аудит | синтез всех секций | Analysis (audit) | только по `verified`-фактам |

## 9. Фазы внедрения

Один спек, реализация фазами; приоритеты пользователя — первыми.

**Фаза 1 — Контакты + финансовые сигналы (приоритет):**
- БД: статус `data_partner`; новые `record_type`.
- Резолверы финансов: `sec_edgar` (Form D), `gov_grants`, `gdelt` (funding).
- Contacts поднимается в основной поток; arXiv-почты + GitHub-почты; tier/обоснование в `info`.
- Relevance: тег `data_partner` вместо отсева.

**Фаза 2 — Глубина анализа + верификация:**
- Стадия Verification + пороги свежести в `config`.
- Полная переписка `analysis_section_task.md` (чек-листы по секциям, claim vs fact).
- Аудит: синтез «деньги + момент + потребность» + угол оффера со ссылкой на ЛПР.
- Новости: дедуп через GDELT DOC + RSS.

**Фаза 3 — Широта источников + полировка:**
- Компания/Продукт: Wikidata, OpenCorporates, Papers with Code, arXiv, детект пивота (Wayback).
- `market_data` для публичных (опционально).
- Monitor → точечный ре-анализ затронутых секций.
- Notion-маппинг новых выводов.

Каждая фаза даёт самостоятельный прирост качества досье.

---

## 10. Вне рамок (YAGNI)

- Платные источники (LinkedIn API, Crunchbase API, SimilarWeb, Apollo, Hunter).
- Переход `dossiers` → `company_profiles` (отдельное решение, не блокирует этот дизайн).
- Конкретные поля/критерии итоговой таблицы досье (пересматриваются позже,
  по запросу пользователя — пока пропущено).
- Субъективные производные оценки в виде типизированных полей (`hot/warm/cold` и т.п.).
- Reverse-sync из Notion.

## 11. Допущения и открытые вопросы

- Пороги свежести 12/6/18 мес приняты по умолчанию; легко меняются в `config`.
- Email-pattern даёт кандидатов без гарантии доставки; «готовность к контакту»
  определяется наличием любого прямого канала, не только email.
- Для `data_partner` пайплайн идентичен трекy покупателя, кроме угла оффера в
  Аудите/Conclusions.
- Открыто: нужен ли `market_data` в Фазе 3 или его отложить полностью (сейчас —
  опционально, для публичных компаний).
- **Открыто (важное):** как именно сохранять партнёрскую принадлежность. Текущее
  решение — статус `data_partner` + durable `research_record` `data_partner_flag`
  (так как статус только повышается). Альтернатива — партнёры идут отдельным треком
  и не продвигаются по статусам покупателя. Нужно подтверждение пользователя.
